# SRS：客户研究需求标准化拆解系统

## 1. 引言

### 1.1 目的
本文档为悦数洞察 Quria 平台的「客户研究需求标准化拆解模块」提供详细软件需求规格说明。

### 1.2 范围
覆盖需求录入、智能拆解、技术翻译、协作确认、需求洞察五个子系统。

### 1.3 定义与缩写

| 缩写 | 全称 |
|-----|------|
| Quria | 悦数洞察 AI 访谈平台 |
| 拆解引擎 | 需求智能拆解引擎 |
| TRD | 技术需求文档（Technical Requirements Document） |
| NLP | 自然语言处理 |

---

## 2. 总体描述

### 2.1 产品视角

```
[客户原始需求] → [需求录入引擎] → [智能拆解引擎] → [技术翻译引擎] → [协作工作台] → [需求确认书]
                                          ↓
                                    [需求洞察看板]
```

### 2.2 用户类型

| 用户类型 | 技术能力 | 使用场景 |
|---------|---------|---------|
| 客户 | 无要求 | 提交需求、查看确认书 |
| 研究员 | 中等 | 录入、拆解、翻译、协调 |
| 技术团队 | 高 | 接收 TRD、批注反馈 |
| 产品经理 | 中等 | 查看洞察看板 |

### 2.3 运行环境
- 服务端：Node.js / Python，与现有 Quria 后端集成
- 客户端：Web 应用，兼容现有 Quria 前端

---

## 3. 功能需求

### 3.1 需求录入引擎（IE）

#### IE-FR-001：文本输入
**描述**：研究员可通过文本框粘贴客户原始需求。  
**输入**：原始需求文本（≤ 5000 字）  
**输出**：结构化提取结果

#### IE-FR-002：语音输入
**描述**：支持上传语音文件或实时录音转文字。  
**输入**：音频文件（MP3/WAV，≤ 30 分钟）  
**输出**：转写文本 + 关键信息提取

#### IE-FR-003：文档上传
**描述**：支持上传 PDF/Word 需求文档。  
**输入**：PDF/DOCX 文件（≤ 10MB）  
**输出**：提取文本 + 关键信息

#### IE-FR-004：关键信息提取
**描述**：自动提取需求中的关键要素。  
**提取字段**：
- 研究目标（必填）
- 目标人群（年龄/性别/职业/地域）
- 研究场景（产品测试/品牌调研/用户画像等）
- 预算范围（如有）
- 时间要求（交付日期）
- 样本量预期

#### IE-FR-005：模糊标记
**描述**：对模糊表述自动标记。  
**规则示例**：
| 模糊表述 | 标记类型 |
|---------|---------|
| "更深入的了解" | 深度不明确 |
| "越多越好" | 数量不明确 |
| "尽快交付" | 时间不明确 |
| "高端用户" | 人群不明确 |

---

### 3.2 智能拆解引擎（DE）

#### DE-FR-001：研究类型判定
**描述**：根据需求内容判定研究类型。  
**输出**：定性 / 定量 / 混合（置信度 ≥ 80%）

#### DE-FR-002：访谈深度分级
**描述**：判定所需访谈深度。  
**级别**：
- L1 筛查：5-10 分钟，快速验证
- L2 探索：15-20 分钟，了解概况
- L3 深度：30-60 分钟，挖掘动机

#### DE-FR-003：样本量推荐
**描述**：基于研究类型和深度推荐样本量。  
**规则**：
| 研究类型 | 深度 | 推荐样本量 |
|---------|------|-----------|
| 定性 | L1 | 20-30 |
| 定性 | L2 | 15-20 |
| 定性 | L3 | 8-12 |
| 定量 | - | 100-300 |

#### DE-FR-004：冲突检测
**描述**：检测需求中的矛盾或不可行组合。  
**冲突规则示例**：
- 样本量 > 50 + 深度 = L3 + 交付时间 < 7 天 → 标红
- 预算 < 1万 + 样本量 > 200 → 标红

#### DE-FR-005：历史数据推荐
**描述**：基于相似历史项目推荐配置。  
**输入**：当前需求特征向量  
**输出**：Top 3 相似项目及其最终配置

---

### 3.3 技术翻译引擎（TE）

#### TE-FR-001：模型配置生成
**描述**：生成 AI 模型配置参数。  
**输出字段**：
- 追问深度模式（精简/标准/深度）
- 行业知识库选择
- 情感探测开关
- 多轮记忆轮数

#### TE-FR-002：Prompt 模板生成
**描述**：生成访谈 Prompt 模板。  
**输出**：系统 Prompt + 追问策略 Prompt + 结束语 Prompt

#### TE-FR-003：数据标签体系
**描述**：生成分析标签框架。  
**输出**：情感标签、主题标签、行为标签、需求标签

#### TE-FR-004：报告框架生成
**描述**：生成报告结构模板。  
**输出**：章节结构 + 可视化建议 + 关键洞察问题清单

#### TE-FR-005：追溯关系建立
**描述**：每条技术需求关联到原始业务需求。  
**格式**：`[技术需求 ID] → [业务需求节点 ID] → [原始需求文本片段]`

---

### 3.4 协作工作台（CW）

#### CW-FR-001：在线批注
**描述**：技术团队可对 TRD 进行批注。  
**批注类型**：疑问 / 建议 / 确认 / 拒绝  
**通知**：批注自动通知研究员

#### CW-FR-002：版本管理
**描述**：记录需求变更历史。  
**功能**：对比版本差异、回滚、变更原因记录

#### CW-FR-003：客户确认
**描述**：生成需求确认书供客户确认。  
**确认方式**：在线确认 / 邮件确认 / 电子签章

---

### 3.5 需求洞察看板（DB）

#### DB-FR-001：高频需求统计
**描述**：展示高频研究类型和场景。  
**维度**：研究类型、行业、客户规模、时间趋势

#### DB-FR-002：模糊表述词云
**描述**：展示最常出现的模糊表述。  
**功能**：点击可查看具体案例和最终澄清结果

#### DB-FR-003：产品化机会标记
**描述**：自动识别可产品化的需求模式。  
**触发条件**：同类需求 30 天内出现 ≥ 3 次  
**输出**：机会描述 + 建议产品化方向 + 预估市场规模

#### DB-FR-004：技术团队反馈分析
**描述**：分析技术团队对需求的反馈类型。  
**维度**：需求不清晰、技术不可行、资源不足、其他

## 4. 非功能需求

### 4.1 性能需求

| 编号 | 需求 | 指标 |
|-----|------|------|
| NF-001 | 文本需求拆解 | ≤ 10 秒 |
| NF-002 | 文档上传解析 | ≤ 30 秒 |
| NF-003 | 语音转写 | 实时倍速 ≤ 2x |
| NF-004 | 看板数据加载 | ≤ 3 秒 |

### 4.2 可用性需求

| 编号 | 需求 |
|-----|------|
| NF-005 | 拆解结果支持一键导出为 PDF/Markdown |
| NF-006 | 模糊标记用黄色高亮，冲突标记用红色高亮 |
| NF-007 | 新手引导覆盖首次使用全流程 |

### 4.3 安全需求

| 编号 | 需求 |
|-----|------|
| NF-008 | 客户需求数据加密存储 |
| NF-009 | 不同客户数据隔离 |
| NF-010 | 操作日志保留 2 年 |

## 5. 接口需求

### 5.1 内部接口

| 接口 | 方法 | 输入 | 输出 |
|-----|------|------|------|
| `/api/v1/intake/submit` | POST | 原始需求文本/文件 | 提取结果、模糊标记 |
| `/api/v1/decompose/run` | POST | 需求 ID | 拆解结果 |
| `/api/v1/translate/generate` | POST | 拆解结果 | TRD 文档 |
| `/api/v1/collab/annotate` | POST | TRD ID、批注内容 | 批注结果 |
| `/api/v1/insight/dashboard` | GET | 时间范围、维度 | 看板数据 |

### 5.2 外部接口
- 复用现有 Quria 用户认证、通知系统
- 对接第三方语音转写服务（如讯飞、阿里云）

## 6. 数据需求

### 6.1 数据存储

| 数据类型 | 存储方式 | 保留周期 |
|---------|---------|---------|
| 原始需求 | 对象存储 + 关系型数据库 | 3 年 |
| 拆解结果 | 关系型数据库 | 3 年 |
| TRD 文档 | 对象存储 | 3 年 |
| 协作记录 | 关系型数据库 | 2 年 |
| 洞察指标 | 时序数据库 | 1 年 |

### 6.2 数据质量
- 所有自动提取字段需保留置信度分数
- 人工修正记录需保留，用于模型迭代
- 需求变更需记录变更原因和审批人

| DB-FR-003 | 产品化机会触发正确 | 模拟测试 |
