## 🧩 项目 1 讲法（AI 追问长度动态优化）

### S（背景）
"现有 AI 访谈的追问问题太长、太机械，受访者容易疲劳、中途退出。我想解决**'问得对'还要'问得短'**这个问题。"

### T（任务）
"目标：根据回答长度、设备、访谈模式，**动态控制追问字数**，并把这个过程**可视化**出来。"

### A（动作）—— 挑 2-3 个深点讲
- **动态字数控制**：≤20 字回答→30 字追问；20-100 字→45 字；>100 字→60 字。移动端再砍 25%。**8 类话题模板**按优先级匹配。
- **疲劳度自适应闭环**（最有技术含量）：我用**三信号加权**算疲劳度——字数递减(40分)+间隔递增(30分)+末轮过短(30分)，≥60 分就让 AI 换话题"放生"受访者。**这是一个反馈回路，不是单向功能**。
- **双视角仪表盘**：受访者侧（柱状波形图 + 半圆疲劳度仪表）+ AI 侧（漏斗 5 层深度分类 + 趋势）。

### R（结果）
- 119 个单测里，节奏分析、深度分类、追问规则都有专门测试组覆盖。
- **闭环逻辑有专门测试**（ai-client.test.ts 的 fatigue-driven fallback，5 个用例验证重度/轻度/正常/缺省/不重复）。

⚠️ **诚实边界**（被追问时主动说）：
- "demo 的疲劳信号是**规则引擎加权**，不是训练模型。真实场景接入访谈日志后，加权逻辑不变，只是数据源换成真实日志——**schema 我已经设计好了**。" 这种自知之明是加分项。

## 🧩 项目 3 讲法（客户需求标准化拆解）

### S（背景）
"销售/客户拿来的需求经常是模糊的——'尽快做越多越好的高端用户调研'。研究员手工拆解耗时、口径不一。"

### T（任务）
"把一段模糊文字，**自动拆解成结构化研究方案**，并量化'这份需求有多模糊'。"

### A（动作）—— 三个递进亮点
1. **正则拆解**：从一句话提取出研究目标、人群（年龄+性别+职业+地区）、场景、类型、深度分级 L1/L2/L3、样本量、周期、追问策略、交付物——**10 个维度**。
2. **需求清晰度评分 0-100**（差异化亮点）：模糊词 -8、冲突 -15、缺失量化 -10，clamp 到 [0,100]。"尽快做越多越好"→ 低分红色；"15 人 14 天"→ 高分绿色。**让'需求质量'第一次可度量**。
3. **专业交付**：一键生成 TRD（5 章节）+ **Gantt 时间线**（4 阶段按周期拆分）+ **风险矩阵**（影响×概率）。从"拆需求"升级到"交付方案包"。

### R（结果）
- 清晰度评分、天数解析、阶段拆分、风险衍生**都是纯函数，全有单测**（rules-decompose.test.ts 34 个用例）。
- **AI 成功路径和本地兜底路径，评分口径完全一致**（同一个 `calcClarityScore` 算）——这个设计点很硬，体现一致性意识。

⚠️ **面试金句**（主动抛）：
> "我把项目 3 做成了**'需求拆解 → 执行 → 质量评估'的闭环**，不是单点工具。这是从'能拆解需求'到'保证研究质量全链路'的系统思维升级。"

## 🔥 5 个"加分项"杀手锏（被问"技术上有什么亮点"时，挑 2 个讲）

### 1. 双层本地兜底，UI 永不报错 ⭐⭐⭐
"所有 AI 调用都带本地规则兜底。DeepSeek 挂了/超时/返回乱码，前端**静默切到本地规则**，用户完全无感。我用 `usedAI: boolean` 标记走了哪条路。**这是生产级容错思维，不是学生 demo**。"

### 2. API Key 永不进前端 bundle
"`DEEPSEEK_API_KEY` 只在 Route Handler 读，不带 `NEXT_PUBLIC_` 前缀，客户端只调自己的 `/api/*`，**密钥永远不出服务端**。这是安全基本功。"

### 3. 服务端成本控制
"多轮对话在服务端 `clampMessages`——只保留最近 10 轮 / 8000 字上限。**故意让客户端超时(35s)比服务端(30s)长**，让服务端自己返回 504。这是真实部署才会考虑的成本意识。"

### 4. 三级降级 + 全局错误边界
"AI 失败 → 服务端非 2xx → 客户端 catch → 本地兜底。加上 Next.js 16 的全局 error boundary，**任何未捕获错误都不白屏**，有重试和返回首页。"

### 5. 回归保护测试思维
"追问规则的测试，我在注释里写明'必须与 page.tsx 内联实现逐字节一致'——重构时测试就是**黄金样本**，防止悄悄改变行为。这是工业级测试意识。"

## ❌ 绝对不要说的话

1. ❌ "我这个项目值 XX 万" —— 接不住，别给自己挖坑。
2. ❌ "用了 AI/大模型" 然后说不清 —— 你用的是 API 调用 + 规则，不是自研模型。
3. ❌ "支持高并发/百万用户" —— 没有后端，没压测过，别说。
4. ❌ "性能很好" 没数据 —— 要说就给具体数字（字数上限、轮次限制）。
5. ❌ 把 demo 说成"已上线产品" —— 部署在 Vercel 不等于商业化。

## 📁 现场演示动线（带面试官看 demo 的顺序）

1. **首页** → 指着三张卡说"这是两个优化项目 + 文档入口"

**总时长控制 5-8 分钟，别超**。演示比口述更有说服力。
