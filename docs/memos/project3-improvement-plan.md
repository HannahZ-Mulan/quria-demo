## ① 需求模糊度评分（Requirement Clarity Score）

### 为什么做
- 现在 P3 只输出「模糊表述标记」列表，缺一个**总览分数**。老板第一眼看不到系统「有多懂需求」。
- memo §三-对 P3 增强-1 原话：「给每个输入需求打一个模糊度分数（0–100），直观展示系统识别模糊需求的能力」。

### 设计
**算法（规则引擎，无需 AI）：**
```
基础分 100，逐项扣分：
  - 模糊词命中：每个 -8 分（如「越多越好」「尽快」「差不多」「专业」）
  - 量词缺失：研究样本/周期无具体数字 -10 分
  - 冲突项：每个 -15 分（已被 conflicts 检出的）
  - 目标人群过于宽泛（如「所有用户」）-10 分
最终 clamp 到 [0, 100]
```
反向：**Clarity Score = 100 − 模糊度**，分数越高越清晰。

**UI：**
- 在拆解结果区顶部新增一张「Clarity Score」仪表卡：
  - 大号渐变数字（0–100，复用现有 `.stat-number` 样式）
  - 一条横向进度条，分段着色：🔴 0–40 模糊 / 🟡 41–70 待澄清 / 🟢 71–100 清晰
  - 文案随分数动态变化（如「⚠️ 需求偏模糊，建议补充：样本量、交付周期」）
- 与现有「模糊表述标记」联动：点击标记可高亮对应扣分项。

### 涉及文件
| 文件 | 改动 |
|------|------|
| `src/lib/rules-decompose.ts` | 新增 `calcClarityScore(raw, result)` 纯函数（易测） |
| `src/lib/types.ts` | `DecomposeResult` 增加 `clarityScore: number` 与 `clarityNotes: string[]` |
| `src/app/api/project3/decompose/route.ts` | 调用算分，填入返回 |
| `src/app/project3/page.tsx` | 新增 `ClarityScoreCard` 组件，置于拆解结果上方 |
| `src/i18n/*.json` | 新增 `clarity_title` / `clarity_vague` / `clarity_fair` / `clarity_clear` 等 key |
| `src/lib/rules-decompose.test.ts` | 为算分函数补单测（边界 0/100、各项扣分） |

### 工期：0.5 天

## ② 访谈质量评分（Interview Quality Score）—— 闭环核心

### 为什么做
- **这是本次改进的叙事核心**。memo §二-Priority 2 强调：P3 负责需求→方案，但缺「回传端」评估执行结果，无法形成闭环。
- 加上它，面试叙事升级为：**「需求拆解 → 执行 → 质量评估 → 反馈优化」**，从单点工具变成系统方法论（memo §四叙事结构第 5 步直接对应）。
- Novel 性：memo 原话「传统研究中访谈质量全靠经验判断，没有量化标准」。

### 设计
**这是个新的轻量「评估」入口**，不破坏现有 4 步对话流，而是在生成 TRD 后追加一步「质量评估」。

**评分维度（4 维，规则引擎 demo）：**
| 维度 | demo 模拟方式 | 权重 |
|------|--------------|------|
| 深度性 Depth | 基于模拟的「追问轮次」「平均回答字数」打分 | 30% |
| 覆盖性 Coverage | 研究目标 → 模拟回答的匹配率（随机 demo 数据） | 30% |
| 自然度 Naturalness | 对话流畅度评分（模拟，1–5 星） | 20% |
| 真实性 Authenticity | 矛盾检测命中率（模拟一致性） | 20% |
综合 = 加权平均 → 0–100。

> **技术边界话术（memo §六）**：「demo 用规则引擎 + 模拟访谈日志数据。真实场景接入访谈日志后，4 个维度的计算逻辑不变，只是数据源换成真实日志——schema 我已设计好。」这同时体现诚实 + 落地意识。

**UI：**
- 在 TRD 步骤后，AI 气泡主动引导「要不要评估这次访谈的执行质量？」
- 点击后弹出 `QualityScorePanel`：
  - 顶部大号总分 + 等级标签（S/A/B/C）
  - **雷达图**（4 维）——这是最出彩的可视化，纯 SVG 手绘即可，无需图表库
  - 每个维度一行：维度名 + 进度条 + 一句解释
  - 底部：「低分项建议」清单（如「覆盖性偏低：目标 3 个，仅深入了 1 个，建议补充追问」）

### 涉及文件
| 文件 | 改动 |
|------|------|
| `src/lib/types.ts` | 新增 `QualityScore { total; depth; coverage; naturalness; authenticity; level; advice[] }` |
| `src/lib/quality-engine.ts` | 新文件，纯函数 `calcQualityScore(simulatedInterviewData)`，输出 4 维分 + 建议 |
| `src/lib/quality-engine.test.ts` | 单测 |
| `src/app/project3/page.tsx` | Step 增加 `"evaluate"`；新增 `QualityScorePanel`（含 SVG 雷达图）+ `RadarChart` 子组件 |
| `src/app/globals.css` | 雷达图轴线/填充配色 |
| `src/i18n/*.json` | `quality_title` / `quality_depth` / `quality_coverage` / `quality_advice` 等 |

### 工期：1 天（雷达图 SVG 占大头，其余复用现有模式）

## ③ TRD 交付升级：Gantt 时间线 + 风险矩阵

### 为什么做
- memo §三-对 P3 增强-3：「生成的 TRD 加上 Gantt 图，让交付物更专业」。
- 当前 TRD 是纯 Markdown 预览（`<pre>`），老板面「交付感」弱。升级后 = 一份带**时间线 + 风险清单**的方案包。

### 设计
在现有 `TrdCard`（导出 MD/PDF/复制）基础上，于 TRD 文本下方追加两个可视化模块：

**A. 项目 Gantt 时间线**
- 数据来自 `DecomposeResult.duration`（预计周期）+ 各阶段拆分（招募/执行/分析/交付，按周期等比）
- UI：横向时间条，4 个阶段用不同色块（青/紫/粉/琥珀），hover 显示天数
- 纯 div + Tailwind，无需图表库

**B. 风险矩阵（Risk Matrix）**
- 从 `conflicts` + `fuzzyMarks` 衍生：每条风险 = 「影响 × 概率」，落在 2×2 或 3×3 矩阵格子里
- UI：小方格矩阵，红/黄/绿着色，点格子看风险详情
- 价值：把 P3 已有的「冲突检测」能力可视化升华

**C.（可选）导出增强**：导出 MD 时把时间线/风险以 Markdown 表格/mermaid 形式一并写入，保持「导出物 = 页面所见」。

### 涉及文件
| 文件 | 改动 |
|------|------|
| `src/app/project3/page.tsx` | `TrdCard` 内新增 `GanttTimeline` + `RiskMatrix` 子组件 |
| `src/lib/rules-decompose.ts` | `generateTrd` 末尾追加阶段拆分 + 风险清单 Markdown 段落 |
| `src/app/globals.css` | Gantt 色块、矩阵格子样式 |
| `src/i18n/*.json` | `gantt_title` / `phase_recruit` / `risk_matrix` 等 |

### 工期：0.5 天

## 四、实施顺序与依赖

```
Day 1 上午  ① 需求模糊度评分（独立、最快出效果，先建立信心）
Day 1 下午  ③ TRD Gantt + 风险矩阵（改动集中在 TrdCard，与①无耦合）
Day 2 全天  ② 访谈质量评分（最重、最有叙事价值，放最后集中做）
            └ 含 SVG 雷达图 + 单测 + i18n
```

- 三项**相互独立**，可并行或按需取舍；若只有 1 天，优先做 **① + ③**（性价比最高）。
- 全程 **规则引擎 + 前端可视化**，不引入新图表库（SVG 手绘），不依赖 AI Key，保证 demo 永远可跑。

## 五、验收标准（Acceptance Criteria）

- [ ] **模糊度**：输入「我们想深入了解用户，样本越多越好，尽快交付」→ Clarity Score 明显低于「调研一线城市 25-35 岁妈妈群体，样本 50，4 周交付」；分数随模糊词数量单调下降；单测覆盖 0/100 边界。
- [ ] **质量评分**：TRD 生成后出现「评估质量」入口；雷达图 4 轴正确渲染；总分 = 4 维加权；低分项有可读建议。
- [ ] **Gantt/风险**：TRD 卡片下方出现时间线（阶段数 = 4，色块按周期比例）+ 风险矩阵（含 conflicts 衍生的红色格子）。
- [ ] **i18n**：9 种语言均能展示，切换语言无中文残留（除文档原文）。
- [ ] **兜底**：断网/AI Key 缺失时，三项均能用本地规则引擎正常出结果，不白屏。
- [ ] **构建**：`npm run build` + `npm run lint` + `npm run test:run` 全绿。
